01 / The gap

The same rules, two different chains.

BIP110 validation runs inside ConnectBlock(). But an ordinary restart loads the existing UTXO chainstate and trusts its persisted tip and validity flags — it never reconnects those blocks. There is no BIP110 equivalent of the SegWit upgrade guard.

The effective treatment of an already-connected block depends on the history of the datadir, not on the block sequence and the node's current rules alone. Two nodes running identical enforcing rules can therefore disagree about the active chain:

NodeDatadir historyResult for the same historical block
Late-upgraded enforcing node Block connected while RDTS was disabled Retained — stays in the active chain
Fresh enforcing node Block first connected while RDTS was enabled Rejected — never enters the chain

Affected transitions

The demonstrated transition is from a build that accepts legacy rules to a BIP110-enforcing build while reusing a compatible datadir. In this checkout, non-enforcing mainnet mode changes DEPLOYMENT_REDUCED_DATA to NEVER_ACTIVE and disables its deadline; enforcing mode restores the hardcoded deployment parameters. Both modes use the same network and database formats, so the later build loads the earlier block index and chainstate.

Directly demonstrated

Datadir reuse between separately compiled RDTS_CONSENT=IMPLICIT and RDTS_CONSENT=UNSUPPORTED_UNSAFE_NO_ENFORCEMENT binaries from the audited commit.

Core and non-enforcing Knots

The affected source history is not Knots-specific. A compatible datadir advanced by Bitcoin Core or by Knots without BIP110 enforcement can accept non-signaling or otherwise BIP110-invalid blocks and enter the same late-upgrade condition. If the activation client later opens it without rebuilding, those applicable blocks may be trusted without ever receiving BIP110 validation. Compatibility across arbitrary versions remains version-dependent.

Other affected histories

Temporarily downgrading to non-enforcing software, re-enabling enforcement later, or otherwise loading a chainstate advanced by software that did not enforce BIP110.

Unaffected history

A node that enforced BIP110 before it first connected every applicable block rejects violations normally and is not induced by this issue to accept them later.

Required preconditions

All four conditions must hold. No malformed database, local file write, or RPC access is required.

Legacy connection. The node connects a block while its software or configuration does not enforce BIP110.
Applicable violation. At that height and ancestry, BIP110 would reject the block for mandatory signaling or an active RDTS rule.
Persisted state. The block obtains enough proof of work to be followed, and the node persists its UTXO chainstate and block-validity flags.
Late enforcement. The operator starts an enforcing client on the same datadir without rebuilding or fully reconnecting the chainstate.

Expected versus actual startup contract

Expected safe behaviorObserved behavior
Prove that every applicable block was connected under the enforcing rules, reconnect it, or refuse startup and require a rebuild/redownload. Startup opens the coins database, accepts its recorded tip and block-index validity, checks six blocks at level 3, and becomes operational without reconnecting inherited blocks.
Current enforcement state and accepted history agree. The node enforces BIP110 on newly connected descendants while continuing from a historical state that those rules could not have produced.
Scope: a concrete late-soft-fork upgrade gap

Persisted Bitcoin block validity is generally not namespaced by software version or by a complete consensus-ruleset identifier, so this shape is not necessarily unique to BIP110. BIP110 makes it concrete because enforcing and explicitly non-enforcing modes share database formats and users may enable enforcement on an existing datadir. This report establishes the BIP110 transition failure; it does not claim that every possible late soft-fork upgrade is promised to recover automatically.

Not a chain split by itself

This defect does not create block B or guarantee a persistent split. A sufficiently-worked legacy-valid / BIP110-invalid block must first exist. What the defect does is prevent a normal late upgrade from reliably healing that disagreement.

It isn't two chains. It's N.

The two-lane picture above understates the damage. There is no single "BlockSlop chain" set against Bitcoin. Each late-upgraded node freezes in whatever BIP110-invalid history its own datadir happened to hold — and different nodes saw different invalid blocks. The enforcing network can therefore shatter into as many incompatible chains as there are distinct pre-upgrade histories, all simultaneously claiming to enforce BIP110.

Why it multiplies

Provenance is per-datadir, not global. Two late-upgraded nodes that buried different invalid blocks each keep their own — and neither will reorg to the other, because both tips are "valid" under their own trusted history.

Why they can't reconverge

A normal reorg only happens toward more-worked chains a node considers valid. Each fragment already treats its buried ancestor as valid, so nothing forces it back to the clean chain — the split is sticky, not transient.

Why it's worse than a classic fork

A classic contentious fork yields two chains people can name and choose between. Here every fragment reports the same deployment as active, so operators believe they agree while their UTXO sets silently disagree.

What bounds N

The count is limited only by the number of distinct legacy-valid / BIP110-invalid histories that different nodes persisted before upgrading — not by any property of BIP110 itself.

05 / Root cause

Persisted validity doesn't say which rules ran.

Five mechanisms combine. The checks live in the right place — but nothing on the startup path forces them to run again over inherited history.

01

Checks live in ConnectBlock()

Mandatory signaling, output-size, and script rules are all enforced during connection — and work when it's actually called.

02

Validity flags are rule-blind

BLOCK_VALID_SCRIPTS is the highest persisted level, but it never records that BIP110 was enabled or which rules applied.

03

The startup guard is SegWit-only

NeedsRedownload() scans for BLOCK_OPT_WITNESS. There is no BIP110 ancestry or provenance scan.

04

Default startup never reconnects

Defaults are checkblocks=6, checklevel=3. Level 3 disconnects blocks; only level 4 reconnects via ConnectBlock().

05

Crash recovery isn't validation

RollforwardBlock() only spends and adds coins during replay — it never runs the BIP110 checks.

=

Result

A late-upgraded node loads chainstate produced under non-enforcing rules and trusts it — the invalid ancestor stays active across restarts.

# default startup verification — src/validation.h DEFAULT_CHECKBLOCKS = 6 DEFAULT_CHECKLEVEL = 3 # disconnects blocks, never reconnects # CVerifyDB::VerifyDB() level 1 → CheckBlock() # context-free only level 3 → disconnect blocks # from a temporary coins view level 4 → ConnectBlock() # the only level that re-runs BIP110 ← not default
Even BLOCK_VALID_SCRIPTS isn't literal proof

BLOCK_VALID_SCRIPTS is the highest persisted validity level, but ConnectBlock() can skip script checks for sufficiently buried ancestors of an assumevalid block and still raise the block to that level. The status never records that BIP110 was enabled or which exact rules ran when the block was connected — so any provenance fix must account for assumevalid shortcuts, and must be evaluated after any ReplayBlocks() recovery, since RollforwardBlock() updates coins without running the checks.

07 / Precedent

SegWit already solved this shape.

BIP141 recognized that persisted validity from older software may be insufficient after an upgrade — and added a coordinated guard. The equivalent BIP110 pieces are absent.

The source comment that names the problem

ConnectBlock() itself documents this exact hazard and points at NeedsRedownload() as the BIP141 remedy — the approach that was never built for BIP110.

src/validation.cpp — Chainstate::ConnectBlock() lines 2728–2736 
// Check it again in case a previous version let a bad block in// NOTE: We don't currently (re-)invoke ContextualCheckBlock() or// ContextualCheckBlockHeader() here. This means that if we add a new// consensus rule that is enforced in one of those two functions, then we// may have let in a block that violates the rule prior to updating the// software, and we would NOT be enforcing the rule here. Fully solving// upgrade from one software version to the next after a consensus rule// change is potentially tricky and issue-specific (see NeedsRedownload()// for one approach that was used for BIP 141 deployment).
Why this is the crux

The general contextual header function carries the same upgrade warning. BIP110's mandatory- signaling check was deliberately split into ContextualCheckBlockHeaderVolatile() and explicitly re-invoked by ConnectBlock(), so it works during a real reconnect. The failure is that ordinary startup never reconnects inherited blocks at all. The code therefore anticipates precisely the upgrade hazard demonstrated here, while only SegWit has a startup guard.

Don't copy the marker verbatim

Note the asymmetry: BLOCK_OPT_WITNESS is set when block data is received (see piece 1), not after ConnectBlock() runs — so it proves witness-aware receipt, not that every consensus check executed. BIP110's rules are ConnectBlock()-only and contextual, so its marker must record successful validation under the applicable rules, not mere receipt. AssumeUTXO makes this sharper still: snapshot activation literally fakes BLOCK_OPT_WITNESS for ancestors — src/validation.cpp:6503-6510 — which a BIP110 guard must not blindly imitate.

09 / Impact

What can and cannot happen.

Severity is left for maintainers to assign. Impact can be high for an affected miner, wallet, or service; exploitability is conditional and, absent an offending block, dormant.

What can happen

  • A user upgrades to enforce BIP110 but unknowingly keeps following a chain that violates it.
  • The enforcing network can fragment into many chains — one per distinct buried invalid ancestor — all reporting the same deployment as active while holding different tips and UTXO sets.
  • Wallet balances and confirmation status can differ between those nodes.
  • A miner or service can keep building on a fork clean nodes reject.
  • Late upgrading fails to deliver the protection the operator intended.

What is not demonstrated

  • No mainnet block is claimed to violate BIP110; no mainnet data was tested.
  • No divergence between two nodes that enforced the same rules from the start.
  • No bypass of ConnectBlock() for newly received blocks.
  • No proof-of-work bypass, database corruption, or remote code execution.
  • No chain split is created by the finding alone; a sufficiently-worked legacy-valid / BIP110-invalid block must first exist.
Why severity is unassigned

Consequences can be high for an affected miner, wallet, or service because it may stay on a branch rejected by clean enforcing nodes. Exploitability has strong preconditions: the violating block must obtain enough proof of work to enter the victim's active chain, and that victim must enable enforcement only after persisting it. Without such a block, the defect remains dormant.

10 / Fix

Prove BIP110 provenance, or fail closed.

A robust fix either makes the supported upgrade boundary explicit and requires revalidation, or persists enough validation provenance to fail closed when a datadir is advanced without enforcement.

Option A

Per-block validation provenance

Add a status bit — e.g. BLOCK_OPT_BIP110_VALIDATED — set only after all applicable BIP110 checks succeed during a real connection. Scan every applicable active-chain ancestor at startup; if any lacks it, fail closed and require reindex. Tie the marker to a fixed, reviewed ruleset.

Option B

Chainstate ruleset attestation

Per-chainstate metadata: a ruleset fingerprint plus the tip it applies through, updated atomically with the coins database's best-block transition. Startup requires the attested tip and ruleset to match the loaded coins tip.

Option C

Explicit unsupported transition

If enabling BIP110 on a datadir that crossed an applicable height isn't meant to be supported, enforce an unconditional rebuild requirement. Automatic refusal based on durable state beats documentation alone.

Per-block provenance design requirements

A marker is safe only if its meaning is precise and durable. Option A must satisfy all of these properties:

Cover every applicable phase. One ancestry-sensitive helper must include both the mandatory-signaling interval and active RDTS transaction/script rules.
Record successful connection. Set provenance only after all applicable checks succeed in a real ConnectBlock() execution.
Exclude weak paths. Never set it in non-enforcing builds, during block receipt, or because BLOCK_VALID_SCRIPTS was already present.
Exclude dry runs. Do not persist provenance for fJustCheck validation that is not becoming a connected chainstate result.
Account for assumevalid. Force applicable BIP110 script checks or withhold the marker whenever a shortcut skipped them.
Bind the ruleset. A bare bit is insufficient if parameters or semantics can change while preserving its meaning; regtest -vbparams makes this concrete.
Scan the complete applicable ancestry. Startup must inspect every active-chain block at which mandatory signaling or active rules apply, not just the recent verification window.
Fail closed with correct recovery. Require -reindex-chainstate -assumevalid=0 when data exists, or full reindex/redownload when pruning removed it.
Run after replay. Evaluate provenance after ReplayBlocks(), because RollforwardBlock() changes coins without executing BIP110 validation.

Side branches

They need not be pre-marked. If a side branch later becomes a reorganization candidate, its blocks pass through ConnectBlock() before entering the active chain and gain provenance only after successful validation.

Status-bit compatibility

Do not repurpose BLOCK_STATUS_RESERVED without a database audit; older databases may retain its previous AssumeUTXO meaning. Prefer a new unambiguous bit or separate metadata.

Why a global boolean is weak

Older or non-enforcing software can advance the datadir without knowing it must invalidate a global flag. Per-block provenance pinpoints which applicable blocks lack enforcing validation.

Unresolved engineering review

Either storage design needs block-index compatibility, atomic flush ordering, multiple-chainstate, downgrade/re-upgrade, and first-upgrade migration review. No fix was prototyped in this audit.

Fix timing matters

If provenance support ships before any BIP110 rule applies on mainnet, no historical block needs it — enforcing nodes mark blocks prospectively and a later transition is detectable without forcing a rebuild. Mainnet was still before that point at the audited height 958,470. This is time-sensitive.

If a fix ships before applicability

Continuously enforcing nodes can mark or attest applicable blocks prospectively, while non-enforcing nodes do not. A later enforcement transition is detectable without rebuilding pre-BIP110 history.

If a fix ships after applicability

Migration must distinguish history genuinely connected by an enforcing release from history inherited from non-enforcing software. The existing database does not reliably encode that distinction, so a one-time reconnect may be unavoidable.

The applicability helper (conceptual)

Both the marker-setting path and the startup scan should share one ancestry-sensitive test for whether BIP110 provenance is required at a block — covering both the mandatory-signaling interval and the active RDTS rules:

bool BIP110ValidationRequiredAt(const CBlockIndex& index, ChainstateManager& chainman) { const auto state = chainman.m_versionbitscache.State( index.pprev, chainman.GetConsensus(), Consensus::DEPLOYMENT_REDUCED_DATA); return DeploymentMustSignalAfter( index.pprev, chainman.GetConsensus(), Consensus::DEPLOYMENT_REDUCED_DATA, state) || DeploymentActiveAt( index, chainman, Consensus::DEPLOYMENT_REDUCED_DATA); } // after a non-fJustCheck connection where every applicable check ran: if (BIP110ValidationRequiredAt(*pindex, m_chainman) && bip110_checks_complete) { pindex->nStatus |= BLOCK_OPT_BIP110_VALIDATED; m_blockman.m_dirty_blockindex.insert(pindex); }

Pruning and AssumeUTXO policy

Missing provenance and missing recovery data are different states. An unpruned node with all applicable blocks can be directed to reconnect. If pruning removed required blocks, a partial scan cannot establish safety and the node must require a full reindex/redownload path.

An AssumeUTXO chainstate must not receive synthesized BIP110 provenance merely because the snapshot base hash commits to header ancestry. Snapshot acceptance needs an explicit policy tying its trust to a BIP110-valid base-chain commitment, or provenance must remain provisional until background validation establishes it. Current snapshot activation synthesizes BLOCK_OPT_WITNESS for ancestors; copying that behavior for BIP110 without a BIP110-specific trust review would recreate the ambiguity.

What does not fully solve it

  • Rechecking only new blocks — already done; the retained ancestor remains.
  • Raising default verification to level 4 for six blocks — the offender may be older.
  • Calling context-free CheckBlock() again — it applies no deployment-dependent rules.
  • Trusting BLOCK_VALID_SCRIPTS — it doesn't encode which soft-fork rules were active.
  • Relying on the operator to remember a manual reindex — without detecting unsafe chainstate.

Required regression coverage

Before a fix is complete, add automated tests for each transition and recovery path:

Signaling boundary

Non-enforcing → enforcing datadir across the mandatory-signaling boundary.

Active output rules

The same transition across active output-size rules.

Active script rule

The same transition with a script-rule violation.

Deep ancestor

An offending ancestor deeper than the default six-block depth.

Clean restart

Normal restart where all applicable blocks have valid provenance — no false-positive rebuild.

Chainstate recovery

-reindex-chainstate -assumevalid=0 reconnects and stops at the last valid ancestor.

Pruned missing data

Applicable history lacks provenance and its block data is unavailable; startup must fail safely.

Interrupted flush

ReplayBlocks() recovery must not silently create or preserve a false attestation.

AssumeUTXO

Cover snapshot activation, trust semantics, background validation, and final chainstate handoff.

Downgrade / re-upgrade

Enforcing state is advanced by non-enforcing software and then loaded by an enforcing client again.

Boundary reorgs

Reorganizations crossing mandatory-signaling, activation, and expiry boundaries.

Pre-applicability history

No false-positive rebuild requirement for blocks before any BIP110 rule applies.

At least one fix regression test should use separate enforcing / non-enforcing binaries rather than only -vbparams. The audit reproducer already supports this through --legacy-bitcoind.

Operational mitigation before a code fix

For an unpruned node that may have connected applicable blocks before enabling BIP110, rebuild the UTXO chainstate through ConnectBlock() with the script-check shortcut disabled:

$ bitcoind … -reindex-chainstate -assumevalid=0

The explicit -assumevalid=0 matters because buried script checks can otherwise be skipped. In this checkout, mainnet's compiled defaultAssumeValid is height 912,683, before the calculated mandatory-signaling range; user-supplied values and future releases still require review. Pruned nodes and AssumeUTXO chainstates may lack the historical data for complete local reconnection and can require a full -reindex / redownload.

A one-time diagnostic can also reconnect every available block:

$ bitcoind … -checklevel=4 -checkblocks=0 -assumevalid=0

This detected the unpruned audit case, but it is expensive, is not a durable provenance mechanism, and stops at missing pruned data. A partial level-4 scan is not proof of complete applicable history.

Remaining work before disclosure or remediation sign-off

The demonstrated divergence does not depend on these follow-ups, but they remain useful:

  • Refresh the height-specific mainnet deployment-state calculation immediately before disclosure or release.
  • Dynamically test AssumeUTXO/background-chainstate behavior.
  • Dynamically exercise an interrupted coins flush followed by ReplayBlocks(); the current replay conclusion is source-derived.
  • Prototype the provenance guard and run the full functional suite.
  • Have a second reviewer independently reproduce the result from this report.

One helper returns which phases applied

The report proposed a conceptual boolean BIP110ValidationRequiredAt(). The updated patch ships something more precise: ReducedDataValidationStatusAt() returns a bitmask of the two phases — signaling and active — so each is tracked independently rather than collapsed into one "applicable" flag.

src/validation.cpp — new static helper + added 
/** BIP110/RDTS consensus-rule phases selected while connecting this block. */static uint32_t ReducedDataValidationStatusAt(const CBlockIndex& block_index,                                        const ChainstateManager& chainman){  const auto deployment{Consensus::DEPLOYMENT_REDUCED_DATA};  const ThresholdState state{chainman.m_versionbitscache.State(      block_index.pprev, params, deployment)};  uint32_t status{0};  if (DeploymentActiveAt(block_index, chainman, deployment))    status |= BLOCK_OPT_REDUCED_DATA_ACTIVE;  if (DeploymentMustSignalAfter(block_index.pprev, params, deployment, state))    status |= BLOCK_OPT_REDUCED_DATA_SIGNAL;  return status;}
Why the two-bit split matters — proven by a new test

A block can be connected while the deployment is ACTIVE but with mandatory signaling not yet enforced. A single "applicable" bit would let that block's active-rule provenance wrongly satisfy a later signaling check. The updated test test_phase_specific_provenance connects 145 non-signaling blocks under ACTIVE rules, upgrades to a signaling-enforcing config, and confirms startup still refuses — the ACTIVE bit cannot stand in for the SIGNAL bit. I ran this case locally; it passes.

One item still open — independently confirmed

The report flagged the assumevalid interaction, and reading the updated code confirms it survives. In ConnectBlock(), the RDTS script rule runs only when fScriptChecks is true (which assumevalid sets false for sufficiently buried ancestors), but the marker at line 3088 is set with no fScriptChecks condition. So during a reindex that trusts an assumevalid checkpoint, a block can receive BLOCK_OPT_REDUCED_DATA_ACTIVE even though its script rule never executed. The output-size rule is safe — it runs unconditionally — so only the script-rule portion of the ACTIVE phase is exposed. The patch's tests avoid this by always recovering with -assumevalid=0. This remains the sharpest open question before the fix is complete.

Concise root cause · BlockSlop

BIP110 adds consensus checks to ConnectBlock(), but the activation client keeps no persisted record tying an existing chainstate to BIP110 enforcement. A late-upgraded node loads and trusts chainstate produced under non-enforcing rules, and default level-3 startup does not reconnect those blocks. A BIP110-invalid ancestor can remain active across ordinary restarts until revalidation or a reorganization removes it — while a fresh node enforcing the same rules rejects the identical block.
